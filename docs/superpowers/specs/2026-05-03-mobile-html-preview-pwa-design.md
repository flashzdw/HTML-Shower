# 手机端纯前端 HTML 预览网站设计文档

## 1. 文档目标

本文档用于沉淀“手机端纯前端 HTML 预览网站”的产品需求与技术实现方案，作为后续实施、评审与范围控制的依据。

## 2. 项目概述

### 2.1 背景

AI 已经可以快速生成 HTML 页面，但普通同学在手机端往往缺少本地开发环境，也不希望依赖额外平台进行上传、部署或运行。因此，需要一个极简、轻量、开箱即用的手机端工具，让用户复制 AI 生成的 HTML 后即可直接预览。

### 2.2 产品定位

这是一个运行在手机浏览器中的纯前端工具网站，核心职责只有一个：让用户在手机上粘贴 HTML 后一键预览效果，并可通过 PWA 能力保存为主屏幕入口。

### 2.3 目标用户

- 需要在手机端查看 AI 生成 HTML 效果的普通用户
- 没有本地开发环境的学生或轻度内容创作者
- 希望使用极简单一工具，而不是完整建站平台的用户

### 2.4 产品目标

- 用户在手机浏览器中打开页面后，可以立即粘贴 HTML 并预览
- 用户无需登录、上传、部署或配置环境
- 用户可将工具添加到手机主屏幕，后续像原生 App 一样使用
- 工具本体支持离线打开，并在离线时继续完成基础预览

### 2.5 非目标

- 不做账号系统
- 不做云端存储
- 不做多人协作
- 不做多文件项目管理
- 不做可视化拖拽编辑器
- 不承诺离线加载用户 HTML 中所有外链资源

## 3. 产品需求文档

### 3.1 核心价值

产品的核心价值是“简化手机端 HTML 预览流程”。用户不需要理解部署、服务器、打包、编辑器或开发者工具，只需要完成“复制 AI 结果、粘贴、预览”这条最短路径。

### 3.2 核心功能

#### 3.2.1 HTML 粘贴输入

- 页面提供一个大尺寸输入框
- 用户可直接粘贴完整 HTML 代码
- 输入框默认面向手机端触控和长文本粘贴优化

#### 3.2.2 一键预览

- 提供一个主操作按钮 `预览`
- 用户点击后，系统立即渲染输入内容
- 预览结果应在当前站点内完成展示，避免跳转到陌生页面

#### 3.2.3 清空重输

- 提供 `清空` 按钮
- 用户可快速移除现有 HTML，重新粘贴新的内容

#### 3.2.4 添加到主屏幕

- 通过 PWA 能力支持用户将网站添加到主屏幕
- 页面内提供简短说明，引导用户完成添加操作
- 需区分 Android Chrome 与 iPhone Safari 的引导差异

#### 3.2.5 离线可用

- 工具自身的静态资源支持离线缓存
- 用户在离线状态下仍可打开该工具并进行基础预览

### 3.3 典型使用流程

#### 3.3.1 首次使用流程

1. 用户在手机浏览器中打开网站
2. 用户阅读一句话说明，理解用途
3. 用户将 AI 生成的 HTML 粘贴到输入框
4. 用户点击 `预览`
5. 页面展示渲染结果

#### 3.3.2 长期使用流程

1. 用户首次使用后将网站添加到主屏幕
2. 用户后续从主屏幕直接打开
3. 用户继续粘贴 HTML 并预览

#### 3.3.3 离线使用流程

1. 用户曾在联网状态下访问过站点并完成静态资源缓存
2. 用户在离线情况下打开主屏幕入口
3. 工具页面成功启动
4. 用户粘贴纯单文件 HTML 后完成预览

### 3.4 页面结构

#### 3.4.1 首页编辑态

- 顶部：产品名
- 副标题：一句话说明产品用途
- 主体：HTML 输入框
- 操作区：`预览`、`清空`
- 辅助区：添加到主屏幕说明

#### 3.4.2 预览态

- 显示 HTML 渲染结果
- 提供返回编辑入口
- 保持操作成本低，避免用户在手机端来回跳转

### 3.5 交互原则

- 极简优先：只保留当前目标必须的功能
- 手机优先：按钮、间距、字号按手机操作习惯设计
- 即时反馈：点击 `预览` 后应尽快显示结果
- 易理解：第一次访问的用户也能立即上手

### 3.6 成功标准

- 用户可在 10 秒内理解网站用途并完成首次预览
- 用户全程无需注册或依赖其他平台
- 用户可通过主屏幕入口快速再次进入网站
- 工具在离线时可正常启动并继续预览基础 HTML 内容

### 3.7 产品边界说明

- “保存”仅指将工具入口保存到主屏幕，不包含保存用户粘贴的 HTML 内容
- 当前版本不默认记忆用户上一次输入内容
- 如果用户 HTML 中依赖远程图片、远程 CSS 或远程 JS，离线状态下这些资源可能无法使用
- 本产品的角色是预览工具，不是在线编辑器或网页托管平台

## 4. 技术实现文档

### 4.1 技术目标

- 保持项目纯前端、无后端依赖
- 优先使用原生浏览器 API
- 控制包体和实现复杂度
- 确保移动端可用性、稳定性和可维护性

### 4.2 技术选型

#### 4.2.1 推荐栈

- `HTML`：页面结构
- `CSS`：移动端样式与响应式布局
- `JavaScript`：交互逻辑与 PWA 注册
- `manifest.json`：PWA 配置
- `Service Worker`：离线缓存

#### 4.2.2 不采用复杂框架的原因

当前产品功能非常聚焦，原生方案足以覆盖全部需求。引入前端框架会增加构建、依赖管理、升级维护与包体体积成本，不符合“轻量、稳定、易维护”的目标。

### 4.3 总体架构

系统由四个层次组成：

- UI 层：负责输入、按钮、说明文案和视图切换
- 预览层：负责将用户 HTML 注入受限容器并渲染
- PWA 层：负责主屏幕安装与离线启动能力
- 安全隔离层：负责约束预览内容对宿主页面的影响

### 4.4 核心实现方案

#### 4.4.1 HTML 输入方案

- 使用 `textarea` 作为输入控件
- 支持粘贴完整 HTML 文本
- 当前版本不做语法高亮和高级编辑能力

#### 4.4.2 预览渲染方案

推荐使用 `iframe.srcdoc` 渲染用户粘贴的 HTML。

选择理由：

- 实现简单，适合纯前端
- 无需上传文件或借助后端生成地址
- 可以在当前页面内直接展示预览
- 与手机端“就地查看”的交互目标一致

#### 4.4.3 安全隔离方案

`iframe` 必须开启 `sandbox`。

推荐原则：

- 仅开放预览所需的最小权限
- 若需要允许脚本运行，可考虑 `allow-scripts`
- 不开启 `allow-same-origin`，避免预览内容与宿主页面获得同源能力

该策略的目的是即使用户粘贴包含脚本的 HTML，也尽可能将影响限制在 iframe 内部，避免直接干扰工具页面本身。

#### 4.4.4 PWA 安装方案

通过 `manifest.json` 声明以下信息：

- 应用名称
- 短名称
- 图标
- 启动地址
- 显示模式
- 主题色与背景色

页面需同时提供安装引导文案：

- Android：提示用户可通过浏览器安装或添加到主屏幕
- iPhone：提示用户通过 Safari 分享菜单中的“添加到主屏幕”完成添加

#### 4.4.5 离线缓存方案

使用极简 `Service Worker` 缓存静态资源。

缓存内容：

- `index.html`
- `style.css`
- `app.js`
- `manifest.json`
- PWA 图标

缓存策略：

- 对应用静态资源采用 `cache-first`
- 不为用户动态粘贴的 HTML 构建复杂缓存逻辑
- 不对外部第三方资源作强依赖或离线保证

### 4.5 建议文件结构

```text
/index.html
/style.css
/app.js
/manifest.json
/sw.js
/icons/
```

### 4.6 数据流

1. 用户在 `textarea` 中粘贴 HTML
2. 点击 `预览` 按钮
3. 前端读取输入内容
4. 通过 `iframe.srcdoc` 写入用户 HTML
5. 浏览器在 iframe 中渲染结果
6. Service Worker 独立负责站点静态资源缓存

### 4.7 离线能力边界

- 能离线打开的是“工具本体”
- 能离线预览的是“纯单文件 HTML”或不依赖外链资源的内容
- 无法天然离线的是用户 HTML 中引用的远程图片、远程字体、远程样式和远程脚本

如需提高离线成功率，产品文案应提示用户尽量使用单文件 HTML。

### 4.8 兼容性策略

- 主目标环境：Android Chrome、iPhone Safari
- 次要环境：其他现代移动浏览器
- 非主目标环境：微信内置浏览器

原因是微信内置浏览器对 PWA 安装与离线能力支持可能受限，不适合作为主承载环境。

### 4.9 风险与约束

#### 4.9.1 安全风险

- 用户可能粘贴带脚本的 HTML
- 若隔离策略不当，可能影响宿主页面稳定性

结论：必须优先处理 iframe sandbox 设计。

#### 4.9.2 性能风险

- 超长 HTML、大量动画或高负载脚本可能导致手机卡顿

当前策略：

- MVP 先不做复杂性能治理
- 如后续出现问题，再增加字符数提示、重载机制或预览前提醒

#### 4.9.3 认知风险

- 用户可能误以为“离线可用”代表所有网页资源都能离线展示

当前策略：

- 在产品说明中明确离线能力边界
- 引导用户使用单文件 HTML

## 5. MVP 范围

首版只包含以下内容：

- HTML 输入框
- `预览` 按钮
- `清空` 按钮
- iframe 预览区
- `manifest.json`
- 基础 `Service Worker`
- 添加到主屏幕说明

## 6. 明确不纳入本期的能力

- 自动保存用户 HTML
- 历史记录
- 本地文件导入导出
- 在线分享链接
- 多文件联调
- 富文本或可视化编辑器
- 账号和云同步

## 7. 实施建议

建议分四步推进：

1. 先完成单页输入与预览能力
2. 再补齐移动端 UI 细节
3. 增加 `manifest.json` 与 `Service Worker`
4. 最后进行真机兼容性与安全验证

## 8. 结论

该项目最适合采用“原生前端 + PWA + iframe 沙箱预览”的极简方案。这样既能满足手机端开箱即用的核心目标，也能在不引入后端与复杂框架的前提下控制成本、降低维护压力，并为后续有限扩展保留空间。
